# TokenDanceCode 架构对标评估

日期：2026-06-09

## 目标

TokenDanceCode 的目标不是复刻 ClaudeCode、Codex 或 OpenCode 的全部复杂度，而是吸收它们对本地 Coding Agent 有长期价值的工程模式：薄 CLI、结构化事件流、统一工具权限管线、JSONL transcript、provider protocol adapter、SDK facade、发布前可验证门禁。

当前基线：`0f631f3 docs(parallel): record wave4 integration`。

Wave 6 文档基线：`8af8ab1`。

## 当前可保留的架构判断

- CLI 保持薄入口：`packages/cli/src/commands.ts` 解析顶层命令，`packages/cli/src/main.ts` 负责 IO-aware handler wiring。CLI main.ts 过大风险已经成为长期架构风险，后续命令新增必须进入 command registry lane 或命令 helper，而不要把 provider、tool execution 或 session 逻辑塞回 CLI。
- Core 是 runtime 边界：`packages/core/src/runtime.ts` 统一 provider turn、tool registry、permission decision、tool result 和 transcript event。
- SDK 是 AgentHub 唯一稳定入口：`TokenDanceCode -> Thread -> run/runStreamed/context`，AgentHub 不直接依赖 core runtime 内部类。
- Provider 按协议拆分：OpenAI Responses、OpenAI Chat Completions / TokenDance Gateway、Anthropic-compatible Messages 分别适配，错误归一化后再暴露。
- 权限与执行分离：permission mode、risk metadata、PowerShell hard deny、remote approval bridge 是独立管线，不应被 SDK 或 AgentHub 绕过。
- Transcript/session 是事实源：JSONL transcript、resume、search、export/prune candidates 支撑 CLI 和 AgentHub 调试面板。

## 复制范围

| 来源模式 | TokenDanceCode 采用方式 |
|---|---|
| ClaudeCode/Codex 的薄 CLI + runtime 分层 | 保持 CLI/SDK/core 三层，不新增云端控制面 |
| 结构化事件驱动 UI | 保持 `TDCodeEvent` 和 AgentHub `run.agent.*` mapper |
| 工具调用统一权限 | 所有 file/patch/shell/git/worktree/subagent tool 走 registry + permission |
| 可恢复 session/transcript | 继续以 JSONL transcript 和 session metadata 做 resume/debug |
| Provider adapter 独立 | 保持 OpenAI/Anthropic/Gateway protocol adapter 和 opt-in smoke |
| OpenCode command metadata registry | 只复制 command id/category/title/aliases/usage/JSON contract，不复制 full-screen palette |
| 发布前 package gate | 保持 dry-run pack、tarball install smoke、manual npm publish gate |

## 明确拒绝或延后

- 拒绝 plugin marketplace：不做插件市场、MCP marketplace、skills 生态。
- 拒绝 app-server daemon：不做云端 team-agent、daemon、remote-control 平台。
- 不做 IDE extension 或 browser bridge。
- 拒绝 native installer：首版不做原生安装器、自动更新器或系统级后台服务。
- 不做复杂 telemetry、feature flag、rollout trace。
- 拒绝 OpenTUI：不复制 OpenCode 的 Effect service graph、DB sync、plugin runtime、OpenTUI/Solid provider tree 或 provider transform 大矩阵。
- 不做生产 OIDC token exchange；TokenDanceID code exchange、JWKS 验证、Hub-local session 由 AgentHub Hub Server 负责。
- 不把 TokenDanceID access token 当 TokenDance Gateway 模型 API key。

## Wave 5 决策

Wave 5 先把对标结论变成可执行工作流，而不是直接扩大功能面。并行 workstream 定义在 `docs/并行推进计划.md`，状态记录在 `docs/并行执行状态.md`，只读 worktree checker 是 `scripts/verify-wave5-worktrees.mjs`。

当前只读审计已经确认：

- TokenDanceCode 当前骨架方向正确，主要风险不是缺少大功能，而是 CLI `main.ts` 继续膨胀、命令/slash 元数据未统一、provider/permission/session 的边界需要更多不变量测试。
- command registry lane 是 CLI 增量的默认落点：每个顶层命令和 slash command 应有稳定 id、category、title、aliases、usage、JSON contract 和 handler wiring，CLI `main.ts` 只做 IO 组装。
- Codex contract/schema drift gate 是 SDK、provider、transcript、AgentHub event mapper 和 CLI JSON 输出的共同门禁；contract 变化必须先由测试捕获，再更新文档和验收清单。
- npm 发布仍应停在本地 gate + release owner 人工发布，不应由 agent 自动 `npm publish`。
- Gateway quickstart 是 preset 和 readiness 检查，不生成 TokenDance API key；TokenDanceID OIDC helper 是 PKCE login URL 启动，不做 token exchange。
- AgentHub SDK contract 已可消费，但生产接入仍由 AgentHub 侧接自己的 event bus、approval store 和 session lifecycle。

优先级：

1. `W24/W25/W30/W31`：架构决策、发布门禁、provider、permission 是稳定基线。
2. `W26/W27`：SDK contract 和 AgentHub fixture 是 AgentHub 消费闭环。
3. `W28/W29/W32/W33`：Gateway/OIDC onboarding、TUI/command registry、session/subagent 是体验和调试能力。

每个实现 lane 必须 TDD、focused tests、`pnpm verify`，最终由协调侧跑 `pnpm release:next:check`。npm 发布仍需单独人工批准。

## Wave 6 架构判断

Wave 6 只读对比了以下 CLI agent 架构入口：

- ClaudeCode 镜像：`reference/claude-code-source/claude-code-main/src/{commands.ts,tools.ts,Tool.ts,query,context,remote}`。
- Codex：`reference/codex/codex-rs/{core,exec,tools,thread-store,cli}`，以及 `codex-rs/{app-server,exec-server,*sandbox*}` 作为反例边界。
- OpenCode：`reference/opencode/packages/{opencode,llm,core,plugin}` 和 `reference/opencode/specs/v2/session.md`。

结论：TokenDanceCode 继续做本地 CLI harness 和 AgentHub 可消费 SDK，不把 Wave 6 变成平台化重写。三个参考仓库都证明长期维护需要清楚的 runtime 边界、事件形态和工具权限入口；它们也证明插件、云任务、全屏 TUI、桌面壳、企业控制面和跨平台强 sandbox 会快速扩大维护面。

### 保留

| 能力 | 采用方式 | 理由 |
|---|---|---|
| 静态 command metadata registry | 继续扩展 `TOP_LEVEL_COMMAND_METADATA`，后续 slash command 也按 id/category/title/usage/aliases/json contract 建表 | ClaudeCode 和 OpenCode 都把命令集中管理；TokenDanceCode 已有顶层 registry，不需要动态 command plugin |
| 结构化事件流 | 保持 `TDCodeEvent`、AgentHub `run.agent.*` mapper、transcript JSONL 和 CLI renderer 同源 | Codex exec JSONL 和 OpenCode LLM event stream 都说明 UI/SDK/日志必须消费同一事件边界 |
| 工具 registry + schema + permission | 保持工具定义、参数 schema、risk metadata、permission decision、execution、result serialization 的单一路径 | ClaudeCode/OpenCode 的工具目录和 Codex tool spec 都指向同一原则：工具调用不能绕过权限和 transcript |
| Provider protocol adapter | 继续维护 OpenAI Responses、OpenAI-compatible Chat/Gateway、Anthropic Messages 三个 adapter | OpenCode 的 protocol 分层适合作为验证方向，但 provider 数量不跟随扩张 |
| Session/transcript 事实源 | 保持 JSONL transcript、session metadata、resume/search/compact helper | Codex thread-store 和 OpenCode session v2 都把 session 行为放在显式结构里，TokenDanceCode 当前 JSONL 足够 |
| Release/package gate | 保持 `contract:check`、`verify`、`pack:check`、`pack:smoke`、人工 npm publish | Codex/OpenCode 都有发布脚本和包边界；TokenDanceCode 只需要本地可重复门禁 |

### 拒绝

| 能力 | Wave 6 结论 | 原因 |
|---|---|---|
| app-server daemon / remote-control 平台 | 拒绝进入 TokenDanceCode | Codex app-server 适合多前端和远程协作，超出本地 CLI harness |
| plugin marketplace / MCP marketplace / skills 生态 | 拒绝首版实现 | ClaudeCode 和 OpenCode 的动态 command/tool/skill/plugin 体系会引入信任、加载、版本和 UI 成本 |
| OpenTUI / full-screen provider tree | 拒绝作为主线 | OpenCode TUI 适合完整产品界面；TokenDanceCode 采用 scrollback-first renderer |
| desktop app / IDE extension / browser bridge | 拒绝进入 Wave 6 | 这些是产品外壳，不是 core runtime 或 SDK contract |
| 企业/云任务/共享控制面 | 拒绝进入 TokenDanceCode | AgentHub 和服务器侧负责协作、队列、身份和权限域 |
| 真实 provider key 自动发现 | 拒绝 | 默认测试和 docs 不读取项目 `.env`，真实 smoke 继续显式 opt-in |

### 延后

| 能力 | 延后条件 | 目前处理 |
|---|---|---|
| Slash command registry | 顶层 registry 稳定后再做 | 只要求后续新增 slash command 不能回塞到 CLI main.ts |
| 层级 instruction discovery | AGENTS/CLAUDE/README 规则更稳定后再做 | 先保持 context builder 的当前摘要输入 |
| Provider history normalization | 三个现有 adapter 出现真实 drift 后再抽象 | 先用 focused provider tests 固定当前协议 |
| 更强 Windows sandbox | 出现可维护的后端和测试资源后再评估 | 继续用 PowerShell hard deny、workspace boundary、approval bridge |
| Session export/prune UI | AgentHub 调试面板提出明确消费需求后再做 | 现在保留只读 metadata、search、compact、resume |

Wave 6 后续实现顺序应先做稳定性和文档对齐：命令 registry 不变量、event/transcript schema drift、provider/tool schema 测试、permission reason 回放、AgentHub SDK package metadata。任何新增能力都必须说明它属于上表的保留项，否则默认拒绝或延后。
