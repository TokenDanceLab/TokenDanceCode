# Tokendance 架构设计文档

日期：2026-06-07

## 1. 设计目标

Tokendance 的架构目标是实现一个类似 Claude Code / Codex CLI 的个人命令行 Coding Agent，同时保持代码结构清晰、可测试、可长期演进。

核心原则：

- 从零实现正式架构，参考 `reference/s20_comprehensive/code.py` 的机制，不继承其单文件教学结构。
- Core Runtime 与 CLI 分离，CLI 只负责输入、命令路由和事件渲染。
- Provider 差异封装在模型适配层，不泄漏到核心运行时。
- 工具调用统一经过注册、校验、权限、执行、审计管线。
- Working context 可以压缩，完整 transcript 永远是事实源。
- 默认 Windows / PowerShell 优先。
- 不做 daemon、MCP、插件、skills、GUI、IDE、多 Agent 常驻团队。
- 代码尽可能简洁，模块边界清楚，避免过度抽象。

## 2. 总体架构

Tokendance 采用 **Core Runtime + CLI Shell** 架构。

```text
tokendance/
  cli/             # 全局命令、交互循环、slash commands、终端渲染
  core/            # agent runtime、turn loop、事件流、会话编排
  models/          # OpenAI / Anthropic provider adapters
  tools/           # 工具注册、schema、tool execution facade
  permissions/     # default/safe/auto/yolo 权限策略
  execution/       # Local/Venv/Conda/Docker/Worktree executors
  storage/         # 全局状态 + 项目状态读写
  context/         # transcript、compact、resume、memory 检索
  tasks/           # task/todo 数据模型和操作
  git/             # git status/diff/worktree/commit 等结构化能力
  agents/          # subagent、reviewer、worktree delegation
  config/          # 配置合并、setup wizard、key 来源
```

顶层数据流：

```text
用户输入
  -> CLI 解析普通消息或 slash command
  -> Core Runtime 创建 turn
  -> ContextBuilder 组装 system prompt / memory / recent messages
  -> Model Adapter 调用 OpenAI 或 Anthropic
  -> Runtime 接收文本流和 tool calls
  -> Tool Registry 找到工具
  -> Permission Engine 判断是否允许
  -> Executor 或 Domain Service 执行
  -> Transcript/Event Log 记录
  -> ToolResult 回喂模型
  -> 没有 tool calls 后输出最终回复
```

硬边界：

- CLI 不直接调用模型。
- CLI 不直接执行工具。
- 模型不能直接执行工具，只能产生 tool call。
- 工具不自行决定最终权限，权限统一由 PermissionEngine 判断。
- shell 命令不裸跑，必须经过 Executor。
- Provider 原始格式不能流入 core 之外。

## 3. Core Runtime

Core Runtime 是 Tokendance 的运行引擎，负责会话状态、上下文构建、模型调用、工具循环、事件输出和 transcript 记录。

Runtime 内部保持 5 个核心组件：

```text
Core Runtime
  ├─ SessionState
  ├─ ContextBuilder
  ├─ TurnRunner
  ├─ ToolOrchestrator
  └─ EventStream
```

### 3.1 SessionState

保存当前会话可恢复状态：

- messages。
- provider / model。
- permission mode。
- work / teach 输出模式。
- todo 状态。
- linked tasks。
- compact summaries。
- transcript id。
- 当前 working directory。
- active subagents。
- current worktree context。

SessionState 是内存态，持久化由 storage 层负责。

### 3.2 ContextBuilder

每次模型调用前统一组装上下文：

- system prompt。
- 项目说明文件。
- global memory。
- project memory。
- compact summary。
- recent messages。
- task/todo state。
- session resume state。
- transcript retrieval results。

ContextBuilder 是唯一能把各类上下文注入模型输入的位置，避免 memory、tasks、CLI、tools 到处拼 prompt。

### 3.3 TurnRunner

负责一次用户输入的完整执行：

```text
接收 user message
-> 记录 user message
-> 调 ContextBuilder
-> 调模型
-> 流式输出 assistant text
-> 如果有 tool calls，交给 ToolOrchestrator
-> tool results 回写
-> 继续调模型
-> 没有 tool calls，结束 turn
```

继续条件以模型响应中是否包含实际 tool call 为准，不只依赖 provider 的 stop reason。

### 3.4 ToolOrchestrator

负责工具调用统一管线：

```text
tool call
-> 查找工具
-> schema 校验
-> 语义校验
-> 权限检查
-> 执行 handler
-> 大结果截断/落盘
-> 审计记录
-> 返回 tool_result
```

工具失败通常返回 error tool_result 给模型，不直接使整个 turn 崩溃。系统级错误交给 Runtime 错误恢复。

### 3.5 EventStream

Runtime 不直接打印，而是输出结构化事件。

事件示例：

- `UserMessageRecorded`
- `AssistantTextDelta`
- `AssistantMessageDone`
- `ToolCallStarted`
- `PermissionRequested`
- `PermissionResolved`
- `ToolCallCompleted`
- `ToolCallFailed`
- `ContextCompacted`
- `TaskUpdated`
- `TodoUpdated`
- `SubagentStarted`
- `SubagentCompleted`
- `ErrorRecovered`
- `TurnCompleted`

CLI 和 TranscriptWriter 都消费同一批事件，保证 UI 展示和历史记录一致。

## 4. 模型适配层

Tokendance 支持 OpenAI 和 Anthropic。Core Runtime 只使用内部统一格式。

核心内部对象：

```text
TDMessage
TDContentBlock
TDToolSpec
TDToolCall
TDToolResult
TDModelResponse
ModelEvent
```

Provider adapter 负责：

- 内部 message -> provider message。
- 内部 tool spec -> provider tool schema。
- provider streaming chunk -> ModelEvent。
- provider tool call -> TDToolCall。
- TDToolResult -> provider 所需 tool result 格式。
- provider error -> 统一错误类型。

统一错误类型：

- `RateLimited`
- `AuthFailed`
- `ModelNotFound`
- `ContextLengthExceeded`
- `ProviderUnavailable`
- `BadRequest`
- `UnknownProviderError`

Provider 原始对象不得泄漏到 Runtime、ToolOrchestrator、Storage 或 CLI。

## 5. 工具系统

工具系统由 ToolRegistry 和 ToolOrchestrator 组成。

每个工具定义：

```text
ToolSpec
  name
  description
  input_schema
  permission_policy
  concurrency_policy
  result_policy
  audit_policy
  handler
```

工具按领域组织：

```text
tools/file.py
tools/patch.py
tools/shell.py
tools/git.py
tools/task.py
tools/todo.py
tools/context.py
tools/memory.py
tools/subagent.py
tools/review.py
```

所有工具必须通过 ToolOrchestrator 执行。工具 handler 可以调用 domain service，但不能绕过权限、审计和 transcript。

结果分三层：

- 给模型的 tool_result：可摘要、可截断、可带 artifact 引用。
- 给 transcript 的完整事实记录：保存完整结果或 artifact 引用。
- 给 CLI 的展示事件：简洁、可读。

并发策略先设计字段，执行初期可串行：

- `serial`
- `parallel_safe`
- `exclusive`

读类工具未来可并发，写文件、patch、shell、git worktree 操作默认串行。

## 6. 权限系统

PermissionEngine 位于 ToolOrchestrator 中，发生在工具真正执行前。

输入：

- tool name。
- tool arguments。
- permission mode。
- workspace root。
- worktree root。
- session approvals。
- tool risk level。
- path analysis。
- command analysis。
- project config rules。

输出：

```text
PermissionDecision
  behavior: allow | deny | ask
  reason
  risk_level
  rememberable
```

权限模式：

- `default`：工作区内读、写、编辑自动允许；危险 shell、破坏性操作、工作区逃逸需要确认。
- `safe`：读自动允许；写、编辑、patch、shell、mutating git 需要确认。
- `auto`：多数工作区内操作自动允许；危险命令、越界、commit 仍需确认或拒绝。
- `yolo`：尽量少问；硬拒绝规则仍生效。

路径权限：

- 所有路径先 canonicalize。
- Windows 下需要大小写归一。
- 检查是否在 workspace 或允许的 worktree 中。
- 检查 symlink / junction escape。

PowerShell 风险分类需要识别：

- `Remove-Item` / `rm` / `del` / `erase`。
- `Set-ExecutionPolicy`。
- `Stop-Process`。
- `Restart-Computer`。
- `Invoke-WebRequest` 下载脚本后执行。
- `iwr ... | iex`。
- 写系统目录。
- 修改注册表、profile、环境变量。
- `git clean -fdx`。
- `git reset --hard`。
- 包管理安装命令，如 `pip install`、`npm install`。

权限决策必须写入 transcript。

## 7. 执行环境

Execution Layer 负责“怎么运行命令”，不负责“是否允许运行命令”。

统一接口：

```text
Executor.run(command, cwd, timeout, env, stdin) -> CommandResult
```

`CommandResult` 包含：

- exit_code。
- stdout_preview。
- stderr_preview。
- stdout_artifact。
- stderr_artifact。
- duration_ms。
- timed_out。
- command。
- cwd。
- shell。

执行后端：

- `LocalExecutor`：默认本地执行。
- `VenvExecutor`：Python 虚拟环境。
- `CondaExecutor`：Conda 环境。
- `DockerExecutor`：可选 sandbox。
- `WorktreeExecutor`：subagent worktree。

Windows PowerShell 是默认 shell。大 stdout/stderr 保存到当前 session 的 `tool-outputs/` artifact 目录。

区分两类命令：

- `shell_command`：模型或用户给出的 PowerShell 字符串，风险较高。
- `structured_command`：内部工具构造的命令，例如 git status，风险较低。

## 8. 存储架构

Tokendance 使用混合存储。

全局目录：

```text
~/.tokendance/
  config.toml
  .env
  sessions-index.json
  memory/
    global.md
  logs/
    tokendance.log
```

项目目录：

```text
<project>/.tokendance/
  config.toml
  instructions.md
  sessions/
    <session-id>/
      session.json
      transcript.jsonl
      compact/
      tool-outputs/
      edits/
      review/
      subagents/
  tasks/
    tasks.jsonl
    task-index.json
  memory/
    project.md
    task/
  worktrees/
    worktrees.json
  logs/
    project.log
```

### 8.1 Session

`session.json` 保存当前可恢复状态：

```json
{
  "session_id": "...",
  "project_path": "...",
  "started_at": "...",
  "updated_at": "...",
  "provider": "openai",
  "model": "...",
  "permission_mode": "default",
  "mode": "work",
  "transcript_path": "transcript.jsonl",
  "latest_summary": "compact/compact-0003.md",
  "recent_event_cursor": 1234,
  "active_task_ids": [],
  "todo_state": [],
  "resume_state": {}
}
```

### 8.2 Transcript

Transcript 是事实源，必须全量保存：

```text
<project>/.tokendance/sessions/<session-id>/transcript.jsonl
```

一行一个事件，带递增 `seq`：

```json
{"seq":1,"type":"user_message","content":"帮我修测试"}
{"seq":2,"type":"assistant_delta","text":"我先看一下测试..."}
{"seq":3,"type":"tool_call","name":"git_status","args":{}}
{"seq":4,"type":"tool_result","status":"ok","preview":"..."}
{"seq":5,"type":"file_edited","path":"src/a.py","patch_artifact":"edits/edit-0001.patch"}
```

大内容不直接塞入 JSONL：

- stdout/stderr 保存到 `tool-outputs/`。
- patch 保存到 `edits/`。
- review report 保存到 `review/`。
- compact summary 保存到 `compact/`。

Transcript 中保存 preview 和 artifact 引用。

### 8.3 Tasks

Task 使用事件流 + materialized index：

- `tasks/tasks.jsonl`：任务事件源。
- `tasks/task-index.json`：当前任务状态索引。

`task-index.json` 可由 `tasks.jsonl` 重建。

### 8.4 原子写与锁

因为存在 subagent 和后台执行，storage 必须支持：

- JSONL append 锁。
- Windows 文件锁。
- `session.json`、`task-index.json` 使用 temp file + atomic replace。
- Artifact 文件名使用 seq 或 UUID，避免冲突。

## 9. ContextBuilder

ContextBuilder 是唯一组装模型输入上下文的位置。

组装顺序：

1. System identity 与工作模式。
2. 权限模式说明。
3. 当前 workspace / git 状态摘要。
4. 项目说明文件摘要。
5. Global memory。
6. Project memory。
7. Session compact summary。
8. Task / todo 状态。
9. 最近消息窗口。
10. 本轮新用户输入。

ContextBuilder 不读取完整 transcript。它只使用 compact summary、recent events 和经过检索得到的 transcript snippets。

上下文预算由 ContextBuilder 统一管理：

- system prompt 预算。
- memory 预算。
- project instructions 预算。
- recent messages 预算。
- tool result 预算。

## 10. Resume 与 Transcript Retrieval

`tokendance` 默认新会话。

`tokendance resume` 和 `/resume` 才恢复历史会话。

恢复时加载：

- session metadata。
- latest compact summary。
- 最近 N 个 transcript events。
- task/todo state。
- memory pointers。
- artifact index。
- active worktree/subagent state。

不会将完整 transcript 全量塞回模型。

Transcript retrieval 工具：

- `search_transcript(query)`。
- `read_transcript_range(start_seq, end_seq)`。
- `read_artifact(path)`。

权限策略：

- 短片段可自动读取。
- 大段历史、完整工具输出、疑似敏感内容需要确认。

## 11. Task 与 Todo 架构

Task 是持久任务图，Todo 是短期执行计划。

Task 状态：

- `pending`
- `in_progress`
- `blocked`
- `completed`
- `cancelled`

Task 可关联：

- session。
- worktree。
- owner。
- dependency。
- task memory。
- validation result。

Todo 状态：

- `pending`
- `in_progress`
- `completed`

Todo 保存在 session state 中，也可关联 task。复杂任务执行时 Agent 应维护 todo，但 todo 不替代 task。

## 12. Git / Worktree / Subagent

Git 是结构化 domain service。

核心能力：

- status。
- diff。
- log。
- branch。
- worktree create/remove/list。
- summarize changes。
- commit suggestion。
- commit with confirmation。

Subagent 类型：

- readonly investigator。
- coding subagent with worktree。
- readonly reviewer。

修改型 subagent 默认创建独立 worktree。Subagent 在自己的 session scope 内运行，返回：

- summary。
- changed files。
- diff。
- validation result。
- transcript reference。

主 Agent 负责整合或丢弃 subagent 结果。

不做持久队友、自治任务认领团队或 message bus 协议。

## 13. Review / Diff / Revert

Tokendance 采用 Claude / Codex 风格：

- 修改直接落盘，但受权限控制。
- 每次写入记录 edit audit 和 patch artifact。
- `/diff` 查看当前改动。
- `/review` 启动只读 reviewer subagent。
- `/revert` 基于 patch artifact 或 git diff 回滚。

Reviewer 不修改文件，只输出 findings。

Revert 必须谨慎：

- 回滚前展示目标。
- 检查文件是否已被后续修改。
- 如果 patch 无法干净反向应用，要求用户确认或给出手动方案。

## 14. CLI 与 Slash Commands

CLI 负责：

- 命令入口。
- 交互输入。
- slash command routing。
- Runtime event rendering。
- 权限 prompt。
- setup wizard。

CLI 不直接调用模型、工具、storage domain service。

Slash commands 分组：

- 会话：`/help`、`/status`、`/new`、`/resume`、`/exit`、`/clear`。
- 模型与配置：`/model`、`/permissions`、`/config`、`/doctor`、`/mode`。
- 任务与计划：`/plan`、`/tasks`、`/todo`。
- 上下文与记忆：`/compact`、`/transcript`、`/memory`。
- 代码变更：`/diff`、`/review`、`/revert`。
- Subagent 与隔离：`/agents`、`/worktree`。

普通用户消息交给 TurnRunner。Slash command 交给 CommandRouter。

## 15. 配置与密钥

配置来源：

1. CLI flags。
2. 项目配置。
3. 全局配置。
4. 默认值。

配置格式使用 TOML。

API key 来源：

1. 当前进程环境变量。
2. 系统 keyring，Windows Credential Manager 优先。
3. `~/.tokendance/.env`。
4. 明文配置，仅用户明确选择后允许。

项目 `.env` 不默认用于 Tokendance provider key。

## 16. 错误恢复与中断恢复

Runtime 应处理：

- provider rate limit：指数退避。
- provider unavailable：重试后清晰失败。
- context length exceeded：reactive compact 后重试。
- max output：请求 continuation。
- tool timeout：返回 error tool_result。
- command failure：记录并返回模型。
- Ctrl+C：保存 session state，允许 resume。

错误恢复事件必须写入 transcript。

## 17. 质量门

Quality Gate 使用项目配置优先、自动发现兜底。

项目配置：

- `test_command`
- `lint_command`
- `typecheck_command`
- `format_command`

自动发现：

- Python：`pytest`、`ruff`、`mypy`。
- Node：`npm test`、`pnpm test`。
- Go：`go test ./...`。
- Rust：`cargo test`。

验证结果记录到 transcript，并在有关时更新 task。

## 18. 架构取舍

明确不采用：

- LangChain / LangGraph 作为核心运行时依赖。
- Local daemon。
- 插件系统。
- MCP。
- 持久多 Agent 团队。
- 全屏 TUI。

保留未来扩展点：

- DockerExecutor。
- 并发 tool execution。
- OpenAI-compatible provider。
- MCP client。
- GUI 或 IDE renderer。

这些扩展不得影响当前核心模块边界。
