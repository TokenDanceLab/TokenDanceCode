# TokenDanceCode 架构对标评估

日期：2026-06-09

## 目标

TokenDanceCode 的目标不是复刻 ClaudeCode、Codex 或 OpenCode 的全部复杂度，而是吸收它们对本地 Coding Agent 有长期价值的工程模式：薄 CLI、结构化事件流、统一工具权限管线、JSONL transcript、provider protocol adapter、SDK facade、发布前可验证门禁。

当前基线：`0f631f3 docs(parallel): record wave4 integration`。

## 当前可保留的架构判断

- CLI 保持薄入口：`packages/cli/src/commands.ts` 解析顶层命令，`packages/cli/src/main.ts` 负责 IO-aware handler wiring。不要把 provider、tool execution 或 session 逻辑塞回 CLI。
- Core 是 runtime 边界：`packages/core/src/runtime.ts` 统一 provider turn、tool registry、permission decision、tool result 和 transcript event。
- SDK 是 AgentHub 唯一稳定入口：`TokenDanceCode -> Thread -> run/runStreamed/context`，AgentHub 不直接依赖 core runtime 内部类。
- Provider 按协议拆分：OpenAI Responses、OpenAI Chat Completions / TokenDance Gateway、Anthropic-compatible Messages 分别适配，错误归一化后再暴露。
- 权限与执行分离：permission mode、risk metadata、PowerShell hard deny、remote approval bridge 是独立管线，不应被 SDK 或 AgentHub 绕过。
- Transcript/session 是事实源：JSONL transcript、resume、search、export/prune candidates 支撑 CLI 和 AgentHub 调试面板。

## 复制范围

| 来源模式 | TokenDanceCode 采用方式 |
|---|---|
| ClaudeCode/Codex 的薄 CLI + runtime 分层 | 保持 CLI/SDK/core 三层，不新增云端控制面 |
| 结构化事件驱动 UI | 保持 `TDCodeEvent` 和 AgentHub `run.agent.*` mapper |
| 工具调用统一权限 | 所有 file/patch/shell/git/worktree/subagent tool 走 registry + permission |
| 可恢复 session/transcript | 继续以 JSONL transcript 和 session metadata 做 resume/debug |
| Provider adapter 独立 | 保持 OpenAI/Anthropic/Gateway protocol adapter 和 opt-in smoke |
| OpenCode 的命令元数据 registry | 只复制 command id/category/title/aliases/usage/JSON contract，不复制 full-screen palette |
| 发布前 package gate | 保持 dry-run pack、tarball install smoke、manual npm publish gate |

## 明确拒绝或延后

- 不做插件市场、MCP marketplace、skills 生态。
- 不做云端 team-agent、daemon、remote-control 平台。
- 不做 IDE extension 或 browser bridge。
- 不做复杂 telemetry、feature flag、rollout trace。
- 不复制 OpenCode 的 Effect service graph、DB sync、plugin runtime、OpenTUI/Solid provider tree 或 provider transform 大矩阵。
- 不做生产 OIDC token exchange；TokenDanceID code exchange、JWKS 验证、Hub-local session 由 AgentHub Hub Server 负责。
- 不把 TokenDanceID access token 当 TokenDance Gateway 模型 API key。

## Wave 5 决策

Wave 5 先把对标结论变成可执行工作流，而不是直接扩大功能面。并行 workstream 定义在 `docs/并行推进计划.md`，状态记录在 `docs/并行执行状态.md`，只读 worktree checker 是 `scripts/verify-wave5-worktrees.mjs`。

当前只读审计已经确认：

- TokenDanceCode 当前骨架方向正确，主要风险不是缺少大功能，而是 CLI `main.ts` 继续膨胀、命令/slash 元数据未统一、provider/permission/session 的边界需要更多不变量测试。
- npm 发布仍应停在本地 gate + release owner 人工发布，不应由 agent 自动 `npm publish`。
- Gateway quickstart 是 preset 和 readiness 检查，不生成 TokenDance API key；TokenDanceID OIDC helper 是 PKCE login URL 启动，不做 token exchange。
- AgentHub SDK contract 已可消费，但生产接入仍由 AgentHub 侧接自己的 event bus、approval store 和 session lifecycle。

优先级：

1. `W24/W25/W30/W31`：架构决策、发布门禁、provider、permission 是稳定基线。
2. `W26/W27`：SDK contract 和 AgentHub fixture 是 AgentHub 消费闭环。
3. `W28/W29/W32/W33`：Gateway/OIDC onboarding、TUI/command registry、session/subagent 是体验和调试能力。

每个实现 lane 必须 TDD、focused tests、`pnpm verify`，最终由协调侧跑 `pnpm release:next:check`。npm 发布仍需单独人工批准。
