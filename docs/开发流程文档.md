# Tokendance 开发流程文档

日期：2026-06-07

## 1. 文档目标

本文档用于规划 Tokendance 的开发过程：如何按照产品功能需求文档和架构设计文档，一步一步搭建出最终产品。

本文档不是新的产品需求，也不改变架构方向。它只回答三个问题：

- 先做什么，后做什么。
- 每个阶段产出什么。
- 每个阶段如何判断可以进入下一阶段。

开发原则：

- 每个阶段都要产出可运行、可测试的结果。
- 先搭核心骨架，再逐步增加能力。
- 每个模块保持职责单一，避免把逻辑堆进一个大文件。
- 默认 Windows / PowerShell 优先。
- 所有长期状态都要有测试和可恢复验证。
- 能用清晰 Python 标准库完成的，不优先引入复杂框架。

## 2. 开发前置约定

### 2.1 仓库初始化

正式开发开始前，应先将当前目录初始化为 Git 仓库，并提交已有文档：

```powershell
git init
git add docs reference
git commit -m "docs: add tokendance product and architecture docs"
```

如果后续决定不把 `reference/` 纳入主仓库，也应先明确记录原因，并保证 PRD、架构设计、开发流程文档被版本管理。

### 2.2 Python 项目结构

项目采用 `src/` 布局：

```text
TokenDanceCode/
  pyproject.toml
  README.md
  src/
    tokendance/
      __init__.py
      cli/
      core/
      models/
      tools/
      permissions/
      execution/
      storage/
      context/
      tasks/
      git/
      agents/
      config/
  tests/
    unit/
    integration/
  docs/
```

### 2.3 推荐基础依赖

基础依赖保持克制：

- `typer`：CLI 命令入口。
- `rich`：终端渲染。
- `pydantic`：内部数据模型和配置校验。
- `tomli-w`：写 TOML 配置。
- `python-dotenv`：读取 `~/.tokendance/.env`。
- `keyring`：系统密钥管理。
- `openai`：OpenAI provider。
- `anthropic`：Anthropic provider。
- `pytest`：测试。
- `pytest-asyncio`：异步测试。

是否引入 `anyio`、`filelock`、`watchfiles` 等依赖，应在对应阶段根据实际需要决定。

## 3. 阶段总览

开发按 12 个阶段推进：

1. 项目脚手架与开发基础设施。
2. 配置、路径和存储基础。
3. EventStream、SessionState 与 Transcript。
4. CLI Shell 与 slash command 框架。
5. 模型抽象与 Mock Provider。
6. OpenAI / Anthropic Provider。
7. Tool Registry、基础工具与权限系统。
8. Core Runtime 与 Agent Loop。
9. ContextBuilder、compact、resume 与 memory。
10. Git、diff、review、revert 与质量门。
11. Task / Todo 系统。
12. Subagent 与 Worktree 隔离。

第 12 阶段完成后，进入产品硬化阶段：Windows 兼容性、安装体验、错误恢复、文档完善、端到端测试。

## 4. 阶段 1：项目脚手架与开发基础设施

### 目标

建立 Python 包结构，使 `tokendance` 命令可以被本地调用，并具备基础测试、格式化和类型检查入口。

### 主要文件

```text
pyproject.toml
README.md
src/tokendance/__init__.py
src/tokendance/cli/main.py
tests/unit/test_imports.py
```

### 实现内容

- 创建 `pyproject.toml`。
- 定义包名、Python 版本、依赖和 console script。
- 创建 `tokendance` 命令入口。
- `tokendance --version` 可以输出版本。
- `tokendance doctor` 可以输出基础环境信息。
- 建立 `tests/unit/`。

### 验收标准

```powershell
python -m pip install -e .
tokendance --version
tokendance doctor
pytest
```

预期：

- 命令可以执行。
- `doctor` 至少显示 Python 版本、操作系统、当前 shell、当前目录。
- 测试通过。

### 实现内容

  - assistant done。
  - tool call started。
  - tool call completed。
  - permission decision。
  - compact event。

### 验收标准

- 创建 session 后能生成目录和 `session.json`。
- 事件写入 `transcript.jsonl` 后顺序正确。

  - 内部 messages -> Anthropic messages。
  - 内部 tool specs -> Anthropic tools。

- core runtime 测试仍可只依赖 Mock Provider。

## 19. 每阶段通用完成标准

每个阶段结束前必须完成：

- 单元测试通过。
- 相关集成测试通过。
- 新增行为写入或更新文档。
- 关键状态写入 transcript 或测试中验证不会丢。
- Windows 路径和 UTF-8 内容至少有一条测试覆盖。
- 如果仓库已初始化 Git，阶段结束后提交一次。

推荐提交格式：

```text
feat: add config and storage foundation
test: cover transcript append behavior
docs: update development flow
```

## 20. 开发节奏建议

推荐每次只推进一个阶段。一个阶段内部如果过大，可以按模块拆小提交，但不要跨阶段混做。

推荐工作顺序：

1. 写测试。
2. 实现最小代码。
3. 跑测试。
4. 手动验证 CLI 行为。
5. 更新文档。
6. 提交。

遇到架构和 PRD 冲突时，先改文档再改代码。

## 21. 风险控制

重点风险：

- 工具执行权限过松。
- Transcript 不完整，导致 resume 和审计不可靠。
- Windows subprocess quoting 出错。
- Provider 格式泄漏进 core。
- ContextBuilder 变成到处拼 prompt。
- Subagent 修改主工作区，破坏隔离。
- Storage 并发写导致 JSONL 或 index 损坏。

控制方式：

- 所有工具都必须走 ToolOrchestrator。
- 所有命令都必须走 Executor。
- 所有写入都必须记录 edit audit。
- 所有长期状态写入都要有测试。
- Subagent coding task 默认只能在 worktree 中运行。
- Provider 单元测试只验证 mapping，不依赖真实 API。
- 真实 provider 测试放 integration，并允许无 key 时跳过。
