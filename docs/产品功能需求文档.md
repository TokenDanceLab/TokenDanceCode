# Tokendance 产品功能需求文档

日期：2026-06-07

## 1. 产品定位

Tokendance 是一个面向个人开发者的全局命令行 Coding Agent。用户可以在任意本地代码仓库中打开终端，输入 `tokendance`，进入一个类似 Claude Code / Codex CLI 的本地开发代理会话。

Tokendance 应该能够协助用户阅读代码、修改文件、运行命令、管理任务、审查 diff、保存完整上下文，并在用户明确要求时恢复历史会话。

Tokendance 的核心目标不是做平台、市场、云服务或团队协作系统，而是做一个可靠、可控、可观察、适合长期自用的本地命令行 Coding Agent。

`codex/ts-refactor` 分支将产品重构为 TypeScript monorepo。旧 Python v0.1 实现作为功能迁移参考保留，但新增能力默认进入 `packages/*`。

## 2. 目标用户

目标用户是希望在本地终端中全局使用 Coding Agent 的个人开发者。

主要使用环境：

- Windows 优先，尤其是 PowerShell。
- 在本地代码仓库中运行。
- 使用 TypeScript / Node.js 实现和分发。
- 深度适配 Git 工作流。
- 首批支持 OpenAI 和 Anthropic 模型。

## 3. 核心目标

- 提供全局 CLI 命令 `tokendance`。
- 默认在当前项目中开启新会话。
- 仅在用户显式执行 `tokendance resume` 或 `/resume` 时恢复历史会话。
- 将 OpenAI 和 Anthropic 作为一等模型供应商。
- 提供类似 Claude Code / Codex CLI 的滚动式终端体验。
- 所有工具调用都经过代码层权限系统，而不是依赖模型自觉。
- 支持精确文本编辑和 patch 两种文件修改方式。
- 将 Git 作为核心能力，而不是普通 shell 命令的附属功能。
- 保存完整 transcript，同时维护可压缩的工作上下文。
- 支持分层记忆、任务系统、显式恢复会话。
- 支持 subagent，并通过 git worktree 隔离修改型子任务。
- 最终产品以 CLI 和可嵌入 SDK 为核心，不做 GUI、Web、IDE 插件。

## 4. 非目标范围

Tokendance 初期不做以下能力：

- MCP 支持。
- 插件系统。
- Skills 目录或可安装 skills。
- 用户自定义 hook 生态、hook marketplace、动态加载插件。Runtime 可以保留少量 deterministic local lifecycle hooks 作为本地审计/自动化出口，但它们不是可扩展插件系统。
- GUI、桌面应用、Web 应用、IDE 插件。
- 常驻多 Agent 团队系统。
- 自动化团队协议、持久队友、消息总线系统。
- Marketplace、账号、云同步、计费或云端项目管理。

这些能力可以作为未来扩展方向，但不属于核心产品需求。

## 5. CLI 入口

Tokendance 安装后必须暴露全局命令：

```powershell
tokendance
```

必要命令行为：

- `tokendance`：在当前目录启动一个新会话。
- `tokendance resume`：恢复当前仓库最近一次可恢复会话。
- `tokendance resume <session-id>`：恢复指定历史会话。
- `tokendance config ...`：查看或修改配置。
- `tokendance doctor`：检查本地安装、模型配置、shell、Git 和项目环境。

最终产品还应支持一次性提示词：

```powershell
tokendance "fix the failing tests"
```

交互式 `tokendance` 仍是第一等入口。

## 6. 交互式 CLI 体验

Tokendance 使用滚动式命令行界面，而不是全屏 TUI。整体体验应接近 Claude Code 和 Codex CLI。

界面需要展示：

- 用户消息。
- Assistant 回复。
- 模型流式输出。
- 工具调用。
- 工具结果。
- 权限确认。
- compact 和 resume 提示。
- task 和 todo 更新。
- Git / diff 摘要。
- 错误与恢复动作。

Agent 运行逻辑必须和终端渲染分离。Runtime 应输出结构化事件，CLI 负责渲染事件。这样未来即使增加其他 renderer，也不需要重写核心 agent loop。

## 7. 输出模式

Tokendance 支持两种输出风格：

- `work`：默认模式。少说多做，输出简洁，聚焦行动、结果、风险和验证。
- `teach`：教学模式。边做边解释原因，适合学习代码库或研究 Agent 行为。

会话内可以切换：

```text
/mode work
/mode teach
```

也可以支持启动参数：

```powershell
tokendance --teach
```

## 8. 模型供应商

Tokendance 必须支持 OpenAI 和 Anthropic。

要求：

- 内部使用 provider-neutral message 格式。
- 为 OpenAI 和 Anthropic 分别实现 provider adapter。
- 统一内部 `ToolCall` 和 `ToolResult` 表示。
- 每个 provider 可配置默认 model。
- 每个 provider 可配置 API base URL。
- 支持模型流式输出。
- 对缺失 key、错误 model、限流、供应商错误给出清晰提示。

Provider 可在全局和项目级配置中分别设置。

## 9. 配置系统

Tokendance 支持三种配置方式：

- 可编辑配置文件。
- CLI 配置命令。
- 首次运行 setup wizard。

配置文件位置：

- 全局配置：`~/.tokendance/config.json`。
- 项目配置：`<project>/.tokendance/config.json`。

配置优先级：

1. CLI flags。
2. 项目配置。
3. 全局配置。
4. 内置默认值。

首次运行向导应配置：

- 默认 provider。
- 默认 model。
- API key 来源。
- 默认权限模式。
- 默认 executor backend。
- 项目本地状态保存偏好。

## 10. 密钥与 API Key

API key 使用混合策略。

读取优先级：

1. 当前进程环境变量：
   - `OPENAI_API_KEY`
   - `ANTHROPIC_API_KEY`
2. 系统 keyring，Windows 下优先 Windows Credential Manager。
3. 可选的 `~/.tokendance/.env`。
4. 配置文件明文 key，仅在用户明确选择并看到风险提示后允许。

Tokendance 默认不应读取项目根目录 `.env` 作为自身 provider key 来源。项目 `.env` 通常属于业务配置，可能包含应用密钥，也可能被误提交。项目 `.env` 支持只能作为显式 opt-in。

## 11. 存储模型

Tokendance 使用混合存储。

全局存储目录：`~/.tokendance/`

包含：

- 全局配置。
- Provider 默认值。
- API key 引用。
- 全局记忆。
- 全局 session 索引。
- 安装元数据。

项目本地存储目录：`<project>/.tokendance/`

包含：

- Sessions。
- Transcripts。
- Compact summaries。
- Tasks。
- Todos。
- Project memory。
- Tool output artifacts。
- Diff / edit audit records。
- Worktree metadata。
- Logs。

用户应能将项目状态重定向到全局目录，也应能关闭项目本地状态写入。

## 12. 会话语义

启动行为：

- `tokendance` 永远启动新会话。
- 默认不加载历史上下文。

恢复行为：

- `tokendance resume` 恢复当前项目最近一次可恢复会话。
- `/resume` 在交互式 CLI 中恢复或选择历史会话。
- 恢复会话时加载 session summary、最近消息窗口、当前 task、todo、memory 指针和 transcript 索引。
- 完整 transcript 可查询，但不会无脑塞回模型上下文。

Session state 至少包含：

- Provider / model。
- 权限模式。
- 当前项目路径。
- Working context summary。
- 最近消息。
- Todo 状态。
- Task 关联。
- Compact summaries。
- Transcript references。
- Tool output references。

## 13. Transcript 与上下文

Tokendance 必须同时做到两件事：

- 完整历史记录不丢。
- 模型工作上下文可控。

要求：

- 完整 transcript 永久保存，除非用户删除。
- Transcript 记录 messages、tool calls、tool results、permission decisions、diffs、errors、retries、compaction events、subagent summaries。
- 工作上下文接近模型上限时自动 compact。
- 用户可通过 `/compact` 手动 compact。
- 每次 compact 都记录压缩范围和摘要保存位置。
- CLI 需要明确提示 compact 事件。
- 用户可通过 `/transcript` 查看完整历史。

Agent 可以自动搜索 transcript。短片段读取可自动进行。读取大段历史、完整工具输出或疑似敏感内容时，需要权限判断或用户确认。

## 14. 上下文压缩

Tokendance 支持自动和手动 compact。

自动 compact：

- 在 working context 接近模型上限时触发。
- 保留最近若干轮对话。
- 总结旧对话。
- 大工具输出不直接重新注入，而是保存引用。
- Compact summary 写入项目存储。

手动 compact：

```text
/compact
```

Compact 后的上下文只是工作表示，不是事实源。事实源是完整 transcript。

## 15. 记忆系统

Tokendance 实现分层记忆。

记忆层：

- Global memory：用户全局偏好和稳定工作习惯。
- Project memory：当前仓库约定、架构说明、测试命令、常见坑。
- Session memory：会话摘要和最近工作状态。
- Task memory：任务内决策、阻塞点、subagent 结论、验证状态。

Memory 和 resume 必须区分：

- Memory 保存长期稳定事实。
- Resume 恢复某次历史工作现场。

默认记忆行为：

- 用户可以手动添加 global / project memory。
- Agent 可以建议保存记忆。
- 默认情况下，自动推断出的长期记忆需要用户确认后才保存。
- 记忆可以通过 `/memory` 查看、添加、删除和确认。

## 16. Agent Loop 与工具调用

Tokendance 需要 provider-neutral agent loop：

1. 构建 system prompt 和 working context。
2. 将 messages 与 tools 发送给模型。
3. 流式展示 assistant 输出。
4. 检测 tool calls。
5. 验证工具输入。
6. 运行权限检查。
7. 通过 executor 执行工具。
8. 追加 tool results。
9. 继续循环，直到没有 tool calls 或触发停止条件。

Loop 应检测实际 tool call block，而不是只依赖 provider stop reason。

所有工具执行都必须事件化、可审计。

## 17. 内置工具

必要内置工具范围：

- 文件读取。
- 文件写入。
- 精确文本编辑。
- Patch 应用。
- 文件搜索 / glob。
- PowerShell 命令执行。
- Git status / diff / log / branch / worktree。
- Task 管理。
- Todo 管理。
- Transcript 搜索和读取。
- Memory 读取、建议、更新。
- Context compact。
- Subagent dispatch。
- Review。

每个工具都需要：

- 类型化输入 schema。
- 输入验证。
- 权限策略。
- 结果大小限制。
- Transcript 日志。
- 清晰错误信息。

## 18. 文件编辑

Tokendance 使用双轨编辑模型：

- `edit_file(path, old_text, new_text)`：适合小范围精确替换。
- `apply_patch(patch)`：适合复杂修改。

所有写入都必须产生审计记录：

- 文件路径。
- 修改前 hash。
- 修改后 hash。
- Patch 或 edit 摘要。
- Tool call id。
- 权限决策。
- 时间戳。

项目内写入根据权限模式决定是否自动允许。工作区外写入必须确认或拒绝。

## 19. 权限模式

Tokendance 支持多权限模式：

- `default`：默认日常开发模式。工作区内读、写、编辑自动允许；危险 shell 命令、破坏性操作和工作区逃逸需要确认。
- `safe`：保守模式。读操作自动允许；写入、编辑、patch、shell 命令需要确认。
- `auto`：更主动。多数工作区内操作自动允许；危险命令和工作区逃逸仍需确认或拒绝。
- `yolo`：尽量少问；硬拒绝规则仍然生效。

权限由代码强制执行，而不是依赖模型提示词。

权限系统必须包含：

- 硬拒绝列表。
- 路径边界检查。
- PowerShell 命令风险分类。
- 每个工具的权限策略。
- 会话级临时授权。
- 权限决策审计日志。

## 20. 执行环境

Tokendance 必须提供 executor 抽象。

默认行为：

- 在当前项目或选定 worktree 中本地执行命令。
- 默认 shell 是 Windows PowerShell。
- 强制 cwd 边界。
- 应用 timeout。
- 记录 stdout、stderr、exit code、duration。

最终产品执行后端：

- `LocalExecutor`：默认本地执行。
- `VenvExecutor`：支持项目 Python 虚拟环境。
- `CondaExecutor`：支持 conda 环境。
- `DockerExecutor`：可选 sandbox backend。
- `WorktreeExecutor`：在 delegated worktree 中执行。

新增 executor backend 不应要求修改核心 agent loop。

## 21. Windows 与 PowerShell 优先

Windows / PowerShell 是第一优先级。

要求：

- 默认执行 PowerShell 命令。
- 使用 `pathlib` 处理路径。
- 文件默认 UTF-8 编码。
- 正确处理 Windows subprocess quoting。
- 识别危险 PowerShell 操作，包括破坏性 `Remove-Item`、修改执行策略等命令。
- 全局配置放在用户目录下。
- keyring 优先支持 Windows Credential Manager。
- 安装后暴露 `tokendance.exe`。
- Worktree 名称必须兼容 Windows 文件系统限制。

跨平台支持可以后续扩展，但 Windows / PowerShell 必须先好用。

## 22. Git 集成

Git 是核心能力。

必要功能：

- 读取 repository status。
- 读取 diff。
- 读取 log。
- 检测当前 branch。
- 总结 working tree changes。
- 创建和删除 worktree。
- 将 task / subagent 与 worktree 绑定。
- 退出或完成任务前提示未提交改动。
- 生成 commit 建议。
- 只有用户确认后才 commit。

Git 能力应尽量使用结构化内部工具，而不是完全依赖任意 shell 命令。

## 23. Review / Diff / Revert

Tokendance 采用类似 Claude / Codex 的工作流：

- 文件修改在权限允许时直接落盘。
- 所有 diff 都记录，便于后续审查。
- 用户可以用 `/diff` 查看改动。
- 用户可以用 `/review` 启动只读 reviewer。
- Reviewer subagent 只报告问题，不直接修改文件。
- 用户可以用 `/revert` 回滚最近 patch 或指定文件。

Review 目标：

- 当前未提交改动。
- 相对 base branch 的 diff。
- 指定 commit。
- 指定文件或路径。

完成任务时，如果存在未审查或未提交改动，Agent 应提示用户。

## 24. Task 系统

Tokendance 使用完整持久任务系统作为核心概念。

Task 字段：

- id。
- title。
- description。
- status。
- owner。
- dependencies。
- blocked reason。
- linked session。
- linked worktree。
- created time。
- updated time。
- completion summary。

Task 状态：

- `pending`。
- `in_progress`。
- `blocked`。
- `completed`。
- `cancelled`。

能力：

- 从对话中创建任务。
- 列出任务。
- 查看任务详情。
- 认领任务。
- 更新任务状态。
- 增加依赖关系。
- 关联 session 和 worktree。
- 保存 task-specific memory。

Todo 和 task 是不同概念。Todo 是 session 或 task 内的短期执行计划。

## 25. Todo 系统

Tokendance 支持类似 TodoWrite 的短期计划系统。

Todo 属性：

- content。
- status。
- linked task。
- timestamp。

Todo 状态：

- `pending`。
- `in_progress`。
- `completed`。

Todo 可以通过 `/todo` 查看，并应作为 session state 保存。复杂编码任务执行时，Agent 应维护 todo。

## 26. Subagent 与 Worktree 隔离

Tokendance 支持 delegated subagents，但不做持久多 Agent 团队。

Subagent 类型：

- 只读调查型 subagent。
- 独立 worktree 中工作的 coding subagent。
- 只读 review subagent。

要求：

- Main agent 可以为边界清晰的任务启动 subagent。
- 只读 subagent 不能修改文件。
- 修改型 subagent 默认在独立 git worktree 中工作。
- Subagent 返回 summary、diff、test results、changed file list。
- Main agent 负责整合、采纳或丢弃 subagent 结果。
- 用户可以通过 `/agents` 查看活跃 subagents。
- 用户可以通过 `/worktree` 查看和管理 worktrees。

Tokendance 不做 always-on teammates、自治团队协议或持久 agent message bus。

## 27. 质量门

Tokendance 支持混合验证策略。

项目配置可定义：

- `test_command`。
- `lint_command`。
- `typecheck_command`。
- `format_command`。

如果未配置，Tokendance 可以推断常见命令：

- Python：`pytest`、`ruff`、`mypy`。
- Node：`npm test`、`pnpm test`。
- Go：`go test ./...`。
- Rust：`cargo test`。

完成有意义的代码修改后，Agent 应尝试或建议运行合适验证。验证命令仍需经过权限和 executor 策略。

验证结果必须记录到 transcript，并在适用时关联 task。

## 28. Slash Commands

必要 slash commands：

会话：

- `/help`：展示命令帮助。
- `/status`：展示当前会话状态。
- `/new`：在当前 CLI 中开启新会话。
- `/resume`：恢复历史会话。
- `/exit`：退出，并提示未完成事项。
- `/clear`：只清空当前可见终端输出，不影响历史。

模型与配置：

- `/model`：查看或切换模型。
- `/permissions`：查看或切换权限模式。
- `/config`：查看当前有效配置。
- `/doctor`：诊断安装和项目环境。
- `/mode`：切换 `work` / `teach`。

任务与计划：

- `/plan`：进入只规划不修改模式。
- `/tasks`：管理持久任务。
- `/todo`：查看当前 todo。

上下文与记忆：

- `/compact`：手动压缩上下文。
- `/transcript`：查看或搜索 transcript。
- `/memory`：管理记忆。

代码变更：

- `/diff`：查看当前改动。
- `/review`：运行只读代码审查。
- `/revert`：回滚最近 patch 或指定文件。

Subagent 与隔离：

- `/agents`：查看和管理 subagents。
- `/worktree`：查看和管理 worktrees。

## 29. 项目说明文件

Tokendance 初期不实现完整 skills 或扩展系统。

但它应读取项目说明文件作为上下文：

- `AGENTS.md`。
- `CLAUDE.md`。
- `README.md`。
- `.tokendance/instructions.md`。
- 相关项目配置文件，如 `package.json`、`pnpm-workspace.yaml`、`tsconfig*.json`、测试和 lint 配置。

项目说明可以影响 Agent 行为，但不能绕过权限策略。

## 30. 可观察性

Tokendance 必须让 Agent 行为可检查、可追溯。

必要记录：

- 完整 transcript。
- Tool call log。
- 权限决策。
- 文件编辑审计记录。
- Diffs。
- 命令输出和 exit code。
- Compact summaries。
- Resume records。
- Subagent summaries。
- Task transitions。
- Errors and retries。

用户应能在事后理解发生了什么，而不是依赖模型记忆。

## 31. 错误恢复

Tokendance 应支持可靠错误恢复：

- Provider 指数退避重试。
- Rate limit 报告。
- 通过 reactive compact 处理上下文溢出。
- 支持 max-output continuation。
- 工具 timeout 处理。
- 命令失败总结。
- 安全中断与 resume。
- 恢复失败时给出清晰错误。

错误恢复不能破坏 transcript 完整性。

## 32. 打包与安装

TokenDanceCode 当前作为 TypeScript/npm monorepo 分发。

要求：

- planned public npm 包为 `@tokendance/code-core`、`@tokendance/code-sdk`、`@tokendance/code-cli`，真实发布前以 release readiness 检查结果为准。
- CLI 包暴露 `tokendance` 命令。
- 兼容 Windows 和 PowerShell 使用路径。
- 支持源码安装后的 `pnpm verify`、`pnpm pack:smoke` 和 `pnpm release:next:check`。
- 支持通过 reviewed npm tarball 执行 first publish，检查脚本不执行 `npm publish`。
- 提供 `tokendance doctor` 和 `tokendance doctor --json` 验证安装状态。

## 33. 验收标准

最终产品达到要求时，应满足：

- 用户可以安装 Tokendance，并在任意仓库运行 `tokendance`。
- 默认启动新会话。
- 可以显式恢复历史会话。
- OpenAI 和 Anthropic provider 都可使用。
- Agent 可以读代码、改代码、应用 patch、审查代码。
- PowerShell 命令执行可用，并受权限系统控制。
- Git status / diff / worktree 等能力可用。
- Tasks 和 todos 能正确持久化。
- 完整 transcript 被保存并可搜索。
- Context compact 能保持工作连续性，同时不删除完整历史。
- Memory 能区分全局偏好、项目约定、会话摘要、任务状态。
- Subagents 可以调查问题，也可以在隔离 worktree 中修改代码。
- `/diff`、`/review`、`/revert` 支持实用代码审查流程。
- Windows / PowerShell 体验是一等能力。
